# 桌面端模块拆分、引用联动与索引进度设计

## 背景与目标

当前 `src/raggg/desktop/main_window.py` 约 1911 行，同时承担 Markdown/LaTeX
渲染、图片定位、窗口布局、收藏夹、会话、来源展示、问答流和索引重建等职责。
本次改动在保持现有界面和行为的前提下完成 B1-B4：

- 将渲染与图片索引逻辑移出主窗口。
- 将收藏夹和会话面板移出主窗口，并使 `main_window.py` 少于 800 行。
- 要求模型以 `[N]` 标注依据，并将标注渲染为可点击上标，联动右侧来源。
- 让知识库构建过程报告阶段进度，并在加载动画下实时显示。

不改变检索排序、索引文件格式、收藏夹 JSON 格式、会话存储格式和主界面视觉主题。

## 模块边界

### `desktop/rendering.py`

提供不依赖主窗口实例的 HTML 渲染能力：

- Markdown、行内 Markdown 和 LaTeX 转换。
- WebView HTML 外壳。
- `[N]` 引用标注转换。
- 图片引用识别和 data URI 转换。

引用只识别正文中的连续方括号编号，例如 `[1]`、`[1][2]`。转换后的链接使用
`RAGGG_CITATION:<N>` 控制台消息通知 Qt 层。编号超出当前来源数量时仍显示为上标，
但不触发来源跳转，避免错误映射。

### `desktop/image_index.py`

封装图片索引构建和从检索来源中提取图片的逻辑。对象负责：

- 扫描 WavEDA 帮助图片及知识库图片目录。
- 规范化图片键。
- 按相对路径、文件名和已知前缀建立映射。
- 根据来源分数和正文图片引用返回候选图片。

主窗口只持有一个图片索引对象，不再包含 `_build_image_index`、
`_add_images_from_root` 等实现。

### `desktop/favorites.py`

封装收藏夹搜索、排序、关键词高亮、文件读写和对话框构建。收藏夹文件路径由主窗口
传入，保存格式保持不变。收藏、删除入口仍由主窗口协调，以便继续更新状态卡片。

### `desktop/sessions.py`

提供会话侧栏组件，持有列表控件并向主窗口发出“新建、切换、删除、收起”信号。
会话数据继续由现有 `SessionManager` 管理；组件不直接生成回答，也不访问 RAG
pipeline。

### `desktop/views.py`

承载纯界面构造和 HTML 视图生成，包括头部、左右面板、聊天区、来源卡片、欢迎页与
空来源页。该模块用于满足主窗口少于 800 行的验收目标，同时使
`WorkbenchWindow` 只保留跨组件协调逻辑。

### `desktop/main_window.py`

保留：

- 应用初始化、组件组装和信号连接。
- pipeline 加载、提问、流式输出和后台 worker 生命周期。
- 索引重建和文件监听协调。
- 引用点击后驱动来源 WebView 滚动。

## 引用数据流

`prompt_builder.build_prompt()` 继续按检索顺序把来源编号为 `[1]`、`[2]`。在最终
提示中新增明确规则：

- 有来源支持的事实句在句末标注对应编号。
- 多个来源写成 `[1][2]`。
- 不得编造不存在的编号。
- 无来源支持的内容不添加引用。

回答经过 Markdown 渲染时，`[N]` 变为可点击的 `<sup>`。点击后聊天 WebView
发送 `RAGGG_CITATION:N`，主窗口调用来源 WebView 的 JavaScript：

1. 找到 `id="source-N"` 的来源卡片。
2. `scrollIntoView()` 将其滚动到可见位置。
3. 临时添加高亮边框，随后恢复。

来源编号始终与 `RAGAnswer.sources` 的显示顺序一致。

## 索引进度数据流

`build_knowledge_base()` 增加可选 `on_progress: Callable[[BuildProgress], None]`
参数。`BuildProgress` 是只读数据对象，包含：

- `stage`：稳定的程序阶段标识。
- `message`：用于 UI 展示的文字。
- `current`、`total`：有自然计数时提供；否则为空。

构建阶段按以下顺序报告：

1. `scan`：扫描 WavEDA 与知识库文档。
2. `chunk`：复用或切分文档。
3. `embed`：复用或生成向量。
4. `save`：写入清单、分块和向量索引。
5. `complete`：构建完成。

回调为空时行为与当前 API 完全一致。桌面后台 worker 将回调转发到已有
`progress` 信号；UI 主线程用信号内容更新 loader 动画下方的文字。手动重建和文件
监听触发的重建都使用同一机制。回调异常不会被静默吞掉，以免掩盖编程错误。

## 错误处理与兼容性

- 抽取模块时保留现有公开导入兼容层；已有测试从 `main_window` 导入的渲染或收藏
  函数仍可工作。
- 图片文件不存在或无法读取时保留原始文本，不阻断整条回答。
- 引用编号不存在时不跳转、不抛异常。
- 构建失败继续走现有 worker 错误信号和提示框；最后一次进度文字可用于判断失败
  阶段。
- 不修改本地用户知识文件，不执行批量删除。

## 测试与验收

先写失败测试，再逐项实现：

- 提示词测试：包含句末引用、多来源格式和禁止虚构编号要求。
- 渲染测试：`[1][2]` 生成两个可点击上标，普通 Markdown 链接或无效文本不被误改。
- 来源联动测试：来源卡片具有稳定 ID，点击消息执行对应滚动脚本。
- builder 测试：阶段按顺序回调，已有无回调调用保持兼容。
- loader 测试：进度信号更新动画下方文字。
- 模块边界测试：新模块可独立导入，原兼容导入仍可用。
- 行数验收：`main_window.py` 严格少于 800 行。
- 回归验收：运行完整测试套件，并以 Qt offscreen 模式实例化主窗口。

最后执行一次真实桌面启动冒烟；如果环境只能完成 offscreen 初始化，将明确区分
“结构/自动化验证通过”和“可见 GUI 人工交互尚未验证”。
